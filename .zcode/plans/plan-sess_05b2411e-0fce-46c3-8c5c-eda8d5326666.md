## 诊断 + 修复方案

### 背景：之前修错方向了
spec 里把"子文件夹打不开"归因为 `g_long_pressed` 标志（长按取消选中后误触发导航）。但：
- 这个修改**已经在工作区里**（`g_local_long_pressed`/`g_remote_long_pressed` 拆分已完成），用户依然报"无论点哪个子文件夹都打不开"
- `g_long_pressed` 只在"长按已选中的文件夹取消选中"后触发，**不可能**解释用户"每次点击都失败"的确定性现象

证据链：
- 客户端：`Local: client/my_ftp` 能正常列出 `Client/`、`Server/`（说明第一层 `opendir("./client")` 和 `opendir("./client/my_ftp")` 成功，CWD 正确）。但点 `Client/` → `opendir("./client/my_ftp/Client")` 返回 NULL → "Unable file"。
- 服务器：`MY_FTP_BOOT="./copy"` 是**相对路径**，`worker_handle_ls` 拼成 `./copy/hahahah/kkk` 再 `opendir`。
- 磁盘上目录确实存在（`client/my_ftp/Client/`、`server/copy/hahahah/kkk/` 都在，权限 `drwxr-xr-x`，名字无隐藏字符 `cat -A` = `Client$`）。

**结论：路径逻辑读起来是对的，目录也存在，所以光读代码无法确定 `opendir` 为什么失败**。强行继续猜 errno 是不负责任的。必须先拿到运行时证据。

### 一个无需运行即可确认的真实 Bug（顺手修）
`client/network_task.c:1090-1100` 的 ST_IDLE 分支只处理 `FTP_CMD_LS && res_result==1`，**静默丢弃 `result==0`**（服务器 opendir 失败时返回的 `"opendir fail"`）。这正好匹配用户服务器端"点 hahahah/kkk 点不开"——列表不刷新也没任何提示。

## 改动清单

### 1. 客户端：诊断日志（`client/ui_manager.c`）
按 spec 原始意图在导航链路加日志：
- `scan_local_directory`（约 1207-1217 行）：打印构造的 `full` 路径、`opendir` 返回值；失败时打印 `errno` + `strerror(errno)`。
- `on_local_file_item_clicked`（约 1503-1517 进入文件夹分支）：打印点击的 `text` 和更新后的 `g_local_cur_path`。
- `on_file_item_clicked`（约 734-748 远程文件夹导航分支）：打印 `text` 和更新后的 `g_remote_cur_path`。

日志走 `fprintf(stderr, ...)`，与现有 `[listdir resp] ...` 风格一致（network_task.c:1075）。

### 2. 客户端：修 LS 失败静默丢弃（`client/network_task.c:1090-1100`）
`FTP_CMD_LS` 分支增加 `else`（`res_result != 1`）：弹错误提示并把 `g_remote_cur_path` 回退到父目录，与本地 `ui_update_local_file_list_cb`（ui_manager.c:1271-1279）的失败处理保持一致。这样服务器 opendir 失败时用户能看到明确反馈，而不是"点不开"。

### 3. 服务器：诊断日志（`server/src/handler.c` `worker_handle_ls` 215 行附近）
`opendir(dir_to_list)` 失败时打印 `dir_to_list` + `strerror(errno)`，验证 `./copy/hahahah/kkk` 在服务器进程 CWD 下是否真实可达（怀疑运行时 CWD ≠ `server/`，导致 `./copy` 解析到空的根 `copy/`，但需日志确认）。

### 4. 重新编译复现 + 共享终端输出
`bash rebuild.sh` → 启动 server/client → 点子文件夹 → 把 stderr 输出贴回来。根据真实的 errno / 路径决定最终修复（可能是：服务器启动目录问题、大小写问题、路径拼接问题，或其他）。这一步在拿到日志后单独处理，不在本次改动内臆断。
